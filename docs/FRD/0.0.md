**Product:** Ziqola - Academic Management System (Multi-Tenant SaaS)  
**Version:** 0.0
**Date:** 2026-03-07

## FR-01: Tenant Registration & Onboarding

### FR-01.1 — Public Registration Form

- The system MUST provide a public-facing registration page for new school tenants.
- The form MUST collect data for school and first user (Admin Staff) with the following required fields:
  - **School:**
    - **Nama Sekolah**: Official school name
    - **Jenjang**: Education level — SD / SMP / SMA / SMK / Lainnya
  - **Admin:**
    - **Nama**: Admin's name
    - **Email**: Admin's email
    - **Password**: Admin's password
- After succesful registration form submission, user will receive a verification email to activate his account.

### FR-01.2 — First login - Onboarding

- Tenant status: `Active`
- Admin Staff User status: `Active`
- Academic Year: Not created
- Curriculum: Not selected
- The system MUST redirect the new Admin Staff to the onboarding page to create first academic year.

## FR-02: User Management

### FR-02.1 — User Identity Model

- The system MUST separate **authentication identity** from **academic identity**.
- All users share base-level attributes: Full Name, Gender, Date of Birth, Email, Phone (optional).
- Academic identifiers (NIP, NUPTK, NIS, NISN) MUST NOT be stored on the base `User` entity — instead, they belong to role-specific profiles (`TeacherProfile`, `StudentProfile`).

### FR-02.2 — Roles

The system MUST support exactly four roles for MVP:

| Role        | Indonesian      | Description                                 |
| ----------- | --------------- | ------------------------------------------- |
| PRINCIPAL   | Kepala Sekolah  | Academic oversight and report card approval |
| ADMIN_STAFF | Tata Usaha (TU) | Administrative operations                   |
| TEACHER     | Guru            | Teaching, attendance, grading               |
| STUDENT     | Siswa           | Read-only academic access                   |

- `HOMEROOM_TEACHER` (Wali Kelas) is NOT a role — it is a **class-level assignment** given to a Teacher for a specific academic year.
- One user MUST belong to exactly one tenant.

### FR-02.3 — Permission Model

- The system MUST use a hybrid RBAC (Role-Based Access Control) model.
- Roles have fixed base permissions with configurable overrides.
- All data access MUST be scoped to the user's `tenant_id`.

## FR-03: Custom Profile Properties

**NOTES**:

- The custom profile properties implementation need to be reviewed again. As with the current implementation is believed to be too complex. Alternative approach is to define fixed custom properties where each tenant can enable/disable and make it required/optional. This way we can avoid the complexity of the current implementation and tenant can safely disable/enable custom properties without the risk of losing data.

### FR-03.1 — Tenant-Specific Custom Fields

- The system MUST support tenant-configurable custom fields for both **Student** and **Teacher** profiles.
- Student and Teacher custom field definitions MUST be **separate** (not shared).
- Only `ADMIN_STAFF` can define, edit, or disable custom fields.
- Custom fields MUST be scoped to `tenant_id`; no cross-tenant access.
- Custom field values are **current-value only** (no historical versioning).

### FR-03.2 — Global Templates

- The system MUST provide versioned **global templates** for Student and Teacher fields.
- Tenants MUST be able to apply these templates or apply customizations.

### FR-03.3 — Tenant Overrides

- After applying a template, tenants MUST be able to: add, edit, or disable fields.
- Disabling a field MUST hide it from forms but MUST retain existing stored values.

### FR-03.4 — Supported Field Types

- Text, Number, Date, Boolean
- Select (single), Multi-select
- File (stored in MinIO, max **5 MB** per file)

### FR-03.5 — Validation Rules

- Required / Optional
- Min/Max (length or numeric value)
- Regex pattern matching
- Date range constraints
- File constraints: size limit and MIME type allowlist

### FR-03.6 — Queryability

- Custom field values MUST be **queryable for filtering** (e.g., filter students by religion, domicile, achievements).
- Data storage MUST support **search and export** of custom field values.

## FR-04: Academic Structure

### FR-04.1 — Tahun Ajaran (Academic Year)

- There MUST be exactly **one Active Academic Year** per tenant at any time.
- Academic Year statuses: `Draft`, `Active`, `Archived`.
- A new Academic Year MUST only be creation-eligible after the **latest Academic Period** of the current year has ended.
- Academic Year date ranges MUST NOT overlap with other Academic Years.
- Activation is **explicit** — users choose to activate a Draft year.
- Each Academic Year contains one or more configurable **Academic Periods**.

### FR-04.2 — Academic Periods

- Academic Periods represent instructional sub-periods within a Tahun Ajaran (e.g., Semester 1, Trimester, Block A).
- Each Academic Period MUST have: Name, Start Date, End Date, Status (`Draft`, `Active`, `Archived`).
- Date ranges MUST NOT overlap within the same Academic Year (including archived periods).
- Each period's start date MUST be after the previous period's end date.
- Period ordering is derived from date ranges (start date → end date).
- Exactly **one Academic Period** may be Active at a time.
- Academic Periods are used consistently by: Schedules, Attendance, Assessments, and Report Cards.

### FR-04.3 — Kelas (Class)

- A Kelas is the **primary academic container** for organizing students and subjects.
- No enforced grade hierarchy — the Kelas structure is flexible.
- Each Kelas MUST have: Name, Academic Year, and optional Grade and Label.
- Classes are unique by `(tenantId, academicYearId, name)`.

**NOTES**: Must be reviewed again aligning with K-12 curriculum structure.

### FR-04.4 — Wali Kelas (Homeroom Teacher) Assignment

- A Kelas MAY have at most **one Wali Kelas per academic year**.
- The Wali Kelas MUST be a user with the `TEACHER` role.
- The assignment is time-bounded and year-scoped.
- A Teacher MAY be Wali Kelas for multiple classes (per school policy).

### FR-04.5 — Rombongan Belajar (Class Group)

- Rombongan Belajar provides optional, flexible classification for classes.
- Supported types: `GRADE`, `STREAM`, `PROGRAM`, `CUSTOM`.
- A class MAY belong to **at most two** Rombongan Belajar:
  - **Exactly one** GRADE-type (if grading is used)
  - **At most one** non-GRADE type (STREAM, PROGRAM, or CUSTOM)
- `GRADE` Rombongan Belajar is **system-managed** (auto-generated from Jenjang).
- Only non-GRADE types are shown in the manual creation UI.
- The system MUST NOT enforce ordering or hierarchy among groups.

**NOTES**: Must be reviewed again aligning with K-12 curriculum structure.

### FR-04.6 — Student Enrollment

- Students MUST be enrollable in a Kelas.
- Class membership MUST be **time-bounded** (supports mid-year transfers).
- Students CAN transfer to another class mid-year by setting an `endDate` on the current enrollment.

## FR-05: Mata Pelajaran & Teaching Assignments

### FR-05.1 — Mata Pelajaran (Subject)

- Subjects are defined **globally per tenant** (not per class or grade).
- A Subject is identified by its base name (e.g., Matematika, Bahasa Indonesia).
- Subjects support soft-delete.

**Globally per tenant reasoning**: If subjects are defined per class or grade (e.g. Matematika X, Matematika XI) we will lose the ability to group them as a single subject (e.g. Matematika). This is not ideal as we might need it later for statistics.

**NOTES**: Must be reviewed again aligning with K-12 curriculum structure.

### FR-05.2 — Teaching Assignment (Class Subject)

- Each **Class–Subject** pair MUST have **exactly one assigned Teacher**.
- Team-teaching is out of scope for MVP.
- The assigned Teacher is responsible for: Attendance, Assessment input, and Descriptive feedback for that class-subject.

### FR-05.3 — Assessment Weights Scope

- Assessment component weights are defined **per ClassSubject + Academic Period**.
- Total weight per ClassSubject per Academic Period MUST equal **100%**.
- Weights MUST be locked once grading starts for that period.

**NOTES**: Assessment weights system should be reviewed again.

## FR-06: Scheduling

**NOTES**: Scheduling implementation should be reviewed again. As of now, the schedule is implemented as it's own entity, but considering we might want to implement custom events, it's to be believed schedule to be implemented as a generic event.

### FR-06.1 — Jadwal Pelajaran (Weekly Schedule)

- The system MUST support **weekly schedules** per Kelas.
- Each schedule slot MUST include: Subject, Teacher (inferred from teaching assignment), Day of Week, Start Time, End Time, Academic Period.
- A Teacher MUST NOT be scheduled in overlapping time slots, even across different classes.
- The system MUST **prevent schedule conflicts** at creation time.

### FR-06.2 — Session Generation

- Admin Staff MUST be able to **manually** create, edit, and delete sessions.
- Teachers MUST be able to **start sessions from the schedule** — the session derives Class, Subject, Teacher, and Period from the Schedule.
- The system MUST prevent **duplicate sessions** for the same schedule and date.
- Teachers MAY start sessions **outside their scheduled time window**.

### FR-06.3 — Automatic Session Generation

- The system MUST support **auto-generation of sessions** on a rolling window (e.g., next 7 days).
- A **nightly cron job** MUST trigger session generation for all tenants with an active Academic Period.
- On schedule create/update, the system MUST **backfill** sessions for the remaining days in the current window.
- Auto-generation MUST be **idempotent** (skips duplicates) and **conflict-aware** (skips time collisions).

**NOTES**: Need to be reviewed again. The current implementation is believed too fragile and not scalable. It should instead be able implemented as a recurrent event. No need for auto-generation of sessions.

## FR-07: Attendance

### FR-07.1 — Attendance Recording

- Attendance MUST be recorded **per Subject Session**, **per Student**.
- Supported attendance statuses:
  - **Hadir** (Present)
  - **Izin** (Excused)
  - **Sakit** (Sick)
  - **Alpha** (Absent / Unexcused)
- One attendance record per `(sessionId, studentProfileId)` — duplicates MUST be prevented.
- Attendance records MUST support an optional `remarks` field.

## FR-08: Assessment & Grading

### FR-08.1 — Assessment Types (Tenant-Configured)

- Assessment Types define categories for assessment components (e.g., Tugas, Ulangan, UTS, UAS).
- Types are **tenant-scoped** and apply to all classes and subjects within the tenant.
- The system MUST provide a **default template** (e.g., Assignment, Quiz, Midterm, Final).
- Types MAY be renamed or disabled by `ADMIN_STAFF`.
- Types MUST NOT be deleted once they are referenced by any assessment component.
- Type identifiers MUST be **immutable** once used in components.

### FR-08.2 — Assessment Components

- Assessment Components are specific graded items per Class–Subject–Period per Type.
- Each Class–Subject combination MUST define its own assessment components.
- Weights across all components per Class–Subject–Period MUST total **100%**.
- Different weighting strategies MAY be used between classes or periods.

### FR-08.3 — Score Input (Nilai Input)

- Teachers MUST be able to input numeric scores per student per assessment component.
- Score editing is allowed until locked.
- Scores support locking (`isLocked`, `lockedAt`).
- A change request workflow MUST exist for teachers to request score corrections after submission.

### FR-08.4 — Assessment Submission Workflow

- Teachers MUST formally submit grades via an assessment submission.
- Once submitted, scores require a change request to be modified.
- Change request statuses: `pending` → `resolved`.

## FR-09: Rapor (Report Card)

### FR-09.1 — Report Card Generation

- Report cards are generated **per student per Academic Period**.
- Each report card MUST include:
  - Numeric score per subject
  - Predikat (grade letter: A, B, C, etc.)
  - Descriptive narrative text (Deskripsi) per subject

### FR-09.2 — Report Card Workflow

1. **Teacher** inputs nilai (scores) and descriptive feedback per subject.
2. **Wali Kelas** (Homeroom Teacher) compiles the report card for the class.
3. **Kepala Sekolah** (Principal) reviews and approves.
4. Report card is **locked** after approval — no further edits allowed.

Report card statuses: `DRAFT` → `REVIEW` → `APPROVAL` → `LOCKED`

### FR-09.3 — PDF Export

- The system MUST support **PDF export** of locked report cards.
